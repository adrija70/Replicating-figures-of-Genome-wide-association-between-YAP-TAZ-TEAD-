The output is the `sam` file which is a format to store the alignment. It is just a text file and you can use `less -S YAP1.sam` to see the content.

After we made one sample work, we can loop over and do the same for other fastq files.

```python
for fq in fastq/*.fastq
do
  bowtie2 \
    -x /home/adrija/projects/genomics/Data/reference/GRCh38_noalt_as/GRCh38_noalt_as \
    -U "$fq" \
    -S "${fq%.fastq}.sam" \
    --threads 8 \
    --no-mixed \
    --no-discordant \
    -k 1
done
```

- What the code means
    
    ## The outer shell loop
    
    ```bash
    for fq in fastq/*.fastq
    do
      ...
    done
    
    ```
    
    This is pure Bash.
    
    - `fastq/*.fastq` expands to **each FASTQ file** in the `fastq/` directory
    - The loop runs once **per file**
    - On each iteration, `fq` holds the full relative path, e.g.
        
        `fastq/sample_01.fastq`
        
    
    This is how you batch-align samples.
    
    ---
    
    ## The Bowtie2 invocation
    
    ```bash
    bowtie2 \
    
    ```
    
    Calls the Bowtie2 aligner. The backslash means “this command continues on the next line”.
    
    ---
    
    ## `x`: the reference index **prefix**
    
    ```bash
    -x /home/adrija/projects/genomics/Data/reference/GRCh38_noalt_as/GRCh38_noalt_as
    
    ```
    
    This is the most misunderstood argument.
    
    - This is **not** a FASTA
    - This is **not** a directory
    - This is the **common filename prefix** of the Bowtie2 index
    
    Bowtie2 internally looks for:
    
    ```
    GRCh38_noalt_as.1.bt2
    GRCh38_noalt_as.2.bt2
    GRCh38_noalt_as.3.bt2
    GRCh38_noalt_as.4.bt2
    GRCh38_noalt_as.rev.1.bt2
    GRCh38_noalt_as.rev.2.bt2
    
    ```
    
    If any of these are missing or mis-capitalized, Bowtie2 aborts.
    
    ---
    
    ## `U`: unpaired (single-end) reads
    
    ```bash
    -U "$fq"
    
    ```
    
    - `U` = **unpaired reads**
    - This tells Bowtie2 the data are **single-end**
    - `"$fq"` is quoted to protect against spaces or weird characters
    
    If this were paired-end data, you would instead use `-1` and `-2`.
    
    ---
    
    ## `S`: SAM output file
    
    ```bash
    -S "${fq%.fastq}.sam"
    
    ```
    
    This is Bash parameter expansion.
    
    - `${fq%.fastq}` removes the `.fastq` suffix
    - Then `.sam` is appended
    
    Example:
    
    ```
    fastq/sample_01.fastq → fastq/sample_01.sam
    
    ```
    
    This keeps outputs aligned with inputs, one-to-one.
    
    ---
    
    ## `-threads 8`
    
    ```bash
    --threads 8
    
    ```
    
    - Uses 8 CPU threads
    - Speeds up alignment
    - Does **not** change results, only runtime
    
    Safe to scale to available cores.
    
    ---
    
    ## `-no-mixed`
    
    ```bash
    --no-mixed
    
    ```
    
    This flag is **irrelevant for single-end data**, but harmless.
    
    Meaning (for paired-end):
    
    - Disallows cases where one mate aligns and the other does not
    
    For your current single-end reads, it has no effect.
    
    ---
    
    ## `-no-discordant`
    
    ```bash
    --no-discordant
    
    ```
    
    Also a paired-end–specific flag.
    
    Meaning:
    
    - Only allow pairs that align with correct orientation and distance
    
    Again: no effect for single-end, but fine to leave in.
    
    ---
    
    ## `k 1`: report only the best alignment
    
    ```bash
    -k 1
    
    ```
    
    This is important.
    
    - Report **only one alignment per read**
    - Even if a read maps equally well to multiple loci, Bowtie2 reports one
    
    This:
    
    - Simplifies downstream analysis
    - Forces **unique reporting**, but not necessarily *unique mappability*
    
    If you wanted all alignments, you would increase `k`.
    
    ---
    
    ## 1. `for`, `do`, `done` — **Bash control structure**
    
    These belong to **Bash**, not Bowtie2.
    
    ```bash
    for fq in fastq/*.fastq
    do
      ...
    done
    
    ```
    
    Read this literally as:
    
    > For each file matching fastq/*.fastq,
    > 
    > 
    > call it `fq`,
    > 
    > and execute the commands between `do` and `done`.
    > 
    - `for` → start a loop
    - `fq` → a **variable name** you chose (arbitrary but conventional)
    - `in fastq/*.fastq` → list of items to loop over
    - `do` → start of loop body
    - `done` → end of loop body
    
    Nothing biological here. Pure shell mechanics.
    
    ---
    
    ## 2. `fq` — **a shell variable**
    
    ```bash
    fq=fastq/sample_01.fastq
    
    ```
    
    At runtime, `fq` holds **one filename at a time**.
    
    On the next iteration, it changes to the next FASTQ file.
    
    The name `fq` is a convention:
    
    - `fq` ≈ FASTQ
    - could just as well be `file`, `x`, or `banana` (please don’t)
    
    ---
    
    ## 3. `$fq` — **variable expansion**
    
    ```bash
    "$fq"
    
    ```
    
    - `$` tells Bash: “replace this with the value of the variable”
    - `$fq` becomes:
        
        ```
        fastq/sample_01.fastq
        
        ```
        
    
    Quoting `"..."` prevents word-splitting and globbing. This is defensive programming.
    
    ---
    
    ## 4. `bowtie2 -x -U -S` — **Bowtie2 options**
    
    Everything from here on is **Bowtie2**, not Bash.
    
    ### `x` — index prefix
    
    ```bash
    -x /path/to/GRCh38_noalt_as
    
    ```
    
    - Tells Bowtie2 **which genome index to use**
    - Points to the *prefix* of `.bt2` files
    - Mandatory
    
    Without `-x`, Bowtie2 has no genome to align against.
    
    ---
    
    ### `U` — unpaired reads
    
    ```bash
    -U "$fq"
    
    ```
    
    - `U` means **single-end FASTQ**
    - The argument is the filename
    - This is where your sequencing reads enter the algorithm
    
    If this is wrong or empty → “No input read files were valid”.
    
    ---
    
    ### `S` — SAM output file
    
    ```bash
    -S sample.sam
    
    ```
    
    - Tells Bowtie2 where to write alignments
    - Output format = SAM (text-based alignment format)
    
    Bowtie2 **does nothing useful** unless `-S` (or piping) is defined.
    
    ---
    
    ## 5. Putting it together in plain English
    
    Your command says:
    
    > For each FASTQ file,
    > 
    > 
    > align its reads (`-U $fq`)
    > 
    > to the GRCh38 reference genome (`-x ...`)
    > 
    > and write the alignments to a SAM file (`-S ...`).
    > 
    
    The shell (`for`, `fq`, `$fq`) handles **iteration and filenames**.
    
    Bowtie2 (`-x`, `-U`, `-S`) handles **biology and alignment**.
    
    ---
    
    ## Mental model (important)
    
    Think in **two layers**:
    
    ### Layer 1: Bash (plumbing)
    
    - `for`
    - `fq`
    - `$fq`
    - `do/done`
    
    ### Layer 2: Bowtie2 (biology)
    
    - `x` → genome
    - `U` → reads
    - `S` → alignments
    
    Most debugging pain comes from mixing these layers in your head.
    
    Once you keep them separate, commands like this stop feeling magical and start feeling mechanical — which is exactly what you want in genomics.
    

Now converting all the sam files to bam files

```python
for sam in fastq/*sam
do
  samtools view -bS $sam > ${sam/sam/bam}
done
```

- What the code means
    
    This snippet is doing **format conversion**—turning Bowtie2’s text output (SAM) into a compressed binary form (BAM). I’ll explain it from the ground up, same two-layer logic as before: **conda → shell → samtools**.
    
    ---
    
    ## 1. Installing the tool
    
    ```bash
    conda install samtools -c bioconda
    
    ```
    
    - `conda install` → install software into your active conda environment
    - `samtools` → a toolkit for manipulating alignment files
    - `c bioconda` → use the Bioconda channel (where bioinformatics tools live)
    
    After this, the command `samtools` becomes available in your shell.
    
    ---
    
    ## 2. The loop structure (Bash again)
    
    ```bash
    for sam in fastq/*sam
    do
      ...
    done
    
    ```
    
    This is the same pattern you already used:
    
    - `for` → start loop
    - `sam` → variable name (holds one SAM filename at a time)
    - `fastq/*sam` → all `.sam` files in the `fastq/` directory
    - `do … done` → commands executed once per file
    
    Example during one iteration:
    
    ```
    sam = fastq/sample_01.sam
    
    ```
    
    ---
    
    ## 3. The core command
    
    ```bash
    samtools view -bS $sam > ${sam/sam/bam}
    
    ```
    
    This line has **three conceptual pieces**.
    
    ---
    
    ### a) `samtools view`
    
    This is the samtools subcommand for **reading and transforming alignments**.
    
    Think of it as:
    
    > “Read an alignment file and rewrite it, possibly changing format.”
    > 
    
    ---
    
    ### b) `bS` flags
    
    ```bash
    -bS
    
    ```
    
    - `S` → input is **SAM** (text format)
    - `b` → output **BAM** (binary format)
    
    So together:
    
    > “Read SAM, output BAM.”
    > 
    
    Modern samtools often auto-detect SAM, but `-S` is explicit and safe.
    
    ---
    
    ### c) `$sam` — input file
    
    ```bash
    $sam
    
    ```
    
    Shell expansion again.
    
    If:
    
    ```
    sam=fastq/sample_01.sam
    
    ```
    
    then samtools receives:
    
    ```
    samtools view -bS fastq/sample_01.sam
    
    ```
    
    ---
    
    ### d) `>` — output redirection
    
    ```bash
    > ${sam/sam/bam}
    
    ```
    
    - `>` is **shell redirection**
    - It sends stdout to a file instead of the terminal
    
    So samtools writes BAM bytes into a file.
    
    ---
    
    ### e) `${sam/sam/bam}` — filename rewriting
    
    This is Bash **string substitution**.
    
    It means:
    
    > “Take the value of $sam and replace the first occurrence of sam with bam.”
    > 
    
    Example:
    
    ```
    fastq/sample_01.sam → fastq/sample_01.bam
    
    ```
    
    So each SAM gets a corresponding BAM with the same base name.
    
    ---
    
    ## 4. What this pipeline does, end to end
    
    In plain language:
    
    > For each SAM file produced by Bowtie2,
    > 
    > 
    > convert it into a BAM file using samtools.
    > 
    
    Nothing more, nothing less.
    
    ---
    
    ## 5. Why BAM matters (biological + practical)
    
    - SAM = human-readable, huge
    - BAM = binary, compressed, indexed
    - All downstream tools (sorting, indexing, variant calling, peak calling) expect BAM
    
    SAM is an intermediate. BAM is the working currency.
    
    ---
    
    ## 6. One important improvement (you *will* want this next)
    
    Right now, BAMs are **unsorted**, which limits their usefulness.
    
    Typical next step:
    
    ```bash
    samtools sort -o sample.sorted.bam sample.bam
    samtools index sample.sorted.bam
    
    ```
    
    But conceptually, you are now past syntax problems and into **pipeline design**, which is the right place to be.
    
    ---
    
    ## Final mental model
    
    - Bash loop → handles files
    - `samtools view` → transforms alignment formats
    - `bS` → SAM → BAM
    - `>` → write to disk
    - `${sam/sam/bam}` → systematic naming
    
    You’re building a clean, linear genomics pipeline. Each step does exactly one job.

**Sort the bam files**

Most of the tools work with the binary form of the sam file which is a bam file. The size of the bam file is smaller `samtools` is the key toolkit to deal with it.
sort the reads in the bam file by coordinates.

```bash
for bam in fastq/*bam
do
  samtools sort -@ 4 -o ${bam/bam/sorted.bam} $bam
done
```

Always test the command for a single sample and then apply it to all samples after you confirm it works.

Why Sort a BAM File?

- Efficient Retrieval by Genomic Coordinates:

Sorting arranges the reads in the BAM file by their genomic coordinates (e.g., by chromosome and position). This is crucial for downstream tools (e.g., variant calling, visualization) that rely on efficient access to reads aligned to specific regions. Compatibility with Analysis Tools:

- Many tools require sorted BAM files as input because they process data regionally (e.g., calling peaks or calculating coverage)
- what the code means
    
    ## The full line
    
    ```bash
    samtools sort -@ 4 -o ${bam/bam/sorted.bam} $bam
    
    ```
    
    This mixes **samtools options** with **Bash variable expansion**. Two layers again.
    
    ---
    
    ## 1. `samtools sort`
    
    This is the samtools subcommand for **sorting alignments**.
    
    Sorting here means:
    
    - Reads are ordered by **chromosome**
    - Then by **leftmost mapping position**
    
    This has nothing to do with read names or quality. It is purely genomic order.
    
    Why this matters:
    
    - Required for indexing (`samtools index`)
    - Required for variant calling, peak calling, coverage, visualization
    
    ---
    
    ## 2. `@ 4` — threads
    
    ```bash
    -@ 4
    
    ```
    
    - Use 4 CPU threads
    - Speeds up sorting
    - Does not affect results
    
    This is samtools’ threading flag (note: different from Bowtie2’s `--threads`).
    
    ---
    
    ## 3. `o` — output file
    
    ```bash
    -o ${bam/bam/sorted.bam}
    ```
    
    - `o` specifies the **output BAM filename**
    - Without it, samtools would write to stdout
    
    The filename is created via **Bash substitution**.
    
    ### `${bam/bam/sorted.bam}` explained
    
    This means:
    
    > Take the value of $bam, replace the first occurrence of bam with sorted.bam.
    > 
    
    Example:
    
    ```
    bam=fastq/sample_01.bam
    
    ```
    
    Then:
    
    ```
    ${bam/bam/sorted.bam} → fastq/sample_01.sorted.bam
    
    ```
    
    So you preserve the base name and add semantic meaning.
    
    ---
    
    ## 4. `$bam` — input BAM file
    
    ```bash
    $bam
    
    ```
    
    Shell expansion again.
    
    This is the **unsorted BAM** produced by `samtools view`.
    
    ---
    
    ## 5. What happens operationally
    
    For one file, this line effectively becomes:
    
    ```bash
    samtools sort -@ 4 -o fastq/sample_01.sorted.bam fastq/sample_01.bam
    
    ```
    
    samtools:
    
    1. Reads the BAM
    2. Sorts alignments by coordinate
    3. Writes a new BAM file
    
    The original BAM remains unchanged.
    
    ---
    
    ## 6. Conceptual summary
    
    In plain English:
    
    > “Using 4 threads, take this BAM file, reorder all alignments by genomic position, and write the result to a new file whose name ends in .sorted.bam.”
    > 
    
    ---
    
    ## 7. Why this step exists at all
    
    Aligners emit reads in **input order**.
    
    Genomic analyses require **coordinate order**.
    
    Sorting is the bridge between:
    
    - “I aligned some reads”
    - “I can now do biology with them”
    
    ---
    
    ## 8. What comes immediately after (for completeness)
    
    Once sorted, you almost always do:
    
    ```bash
    samtools index fastq/sample_01.sorted.bam
    
    ```
    
    Which builds a `.bai` index so tools can jump to genomic regions.
    
    ---
    
    ## Mental anchor
    
    - `samtools view` → format conversion
    - `samtools sort` → genomic ordering
    - `samtools index` → random access
    
    This line is the middle, indispensable step.
